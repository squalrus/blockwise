## API (`apps/api/src/app.ts`)

Auth gates:

- **public** — no auth, or `attachOptionalAuthUser` (personalizes response if a token is present, never blocks)
- **auth** — `requireAuthUser` (any signed-in account)
- **business** — `requireBusinessAccount` (signed in + `account_type = 'business'`)
- **venueOwner** — `requireVenueOwner` (approved claim on the venue at `:id`)
- **admin** — `requireAdmin` (admin of *at least one* neighborhood, no `:id` to scope by — list-style routes only)
- **neighborhoodAdmin** — `requireNeighborhoodAdmin` (admin of the *specific* neighborhood at `:id`)

## History

- **2026-07-16** — Neighborhood-admin Events tab split out of Overview (BACKLOG.md Ref 78, imported from the same Claude Design mockup as the 2026-07-13 sidebar redesign, project "Spored: Mycelial Network Design" → `Spored Admin.dc.html`): new `admin/neighborhood/[neighborhoodSlug]/events/page.tsx` with "Calendar feeds" (`IcalFeedForm`) and "Create event" (`EventForm`) cards on the left and an "Upcoming" list on the right; Overview no longer renders any events UI, though it still fetches the same `GET .../dashboard` summary (also carries description/social_links). Sidebar (`layout.tsx`) gained an "Events" nav entry (no count badge yet — would need its own fetch). New shared `apps/web/src/app/EventListItem.tsx` (a date badge, title, time/venue/location, and a feed-vs-manual source pill, with the description behind a click-to-expand instead of always shown) used identically by this tab, the public Upcoming events tab, and the public Today tab's events section — `actions` (Hide/Unhide + Delete) is admin-only, and `showSource` is false on both public pages since visitors have no use for the feed/manual distinction. Also: the neighborhood profile's default tab was renamed "Happening now" → "Today" (`NeighborhoodTabs.tsx`), and its events section broadened from strictly-in-progress to anything happening today — `HappeningNow.live_events` renamed to `today_events`, and `happeningNow.ts`'s `isLiveNow` (a strict `now` between start/end check) replaced with `isToday` (event's range overlaps `now`'s local calendar day). The route path (`GET /neighborhoods/:id/happening-now`) and file names were deliberately left as-is — only the UI label, the response field, and the filter logic changed. Because `listUpcomingEventsForNeighborhood` already excludes events whose `end_time` has passed (server-side), "today" in practice means "today, and not yet over" — an event earlier today that already ended won't resurface.
- **2026-07-16** — iCal/webcal event feed import (BACKLOG.md Ref 30, still in progress on this branch): a neighborhood or a claimed business can publish an external `.ics`/webcal calendar feed URL, then trigger a manual "Sync now" that fetches and upserts it into the `event` table instead of re-keying every event by hand (EventForm's manual-entry path stays the fallback). `ical_feed_url`/`ical_synced_at` live on `neighborhood` directly, but on `business_claim` (not `venue`) for the venue side — same reasoning as `social_links`, since it's owner-authored business-profile data edited via the approved-claim path, not synced from Google Places. `event` gained `source` ('manual' | 'ical') and `external_uid` (the feed's own iCalendar UID, null for manual rows), with plain per-owner unique constraints on `(neighborhood_id, external_uid)`/`(venue_id, external_uid)` so a re-sync upserts instead of duplicating — NULL-column semantics mean manual rows and the other owner type are automatically exempt. Recurring `RRULE` events are expanded into individual occurrences (via `node-ical`'s `expandRecurringEvent`) within a bounded 180-day/25-occurrence window, each given a synthetic per-occurrence UID. New `PATCH .../ical-feed` and `POST .../ical-feed/sync` routes under both `/business/venues/:id/*` and `/neighborhood-admin/neighborhoods/:id/*`; both dashboard GETs gained `ical_feed_url`/`ical_synced_at`. No scheduler wired up yet — sync is manual-trigger only for now. Also: `listEventsForNeighborhoodAndVenues` (backs both the public Upcoming events tab and Happening now) now excludes events that have already ended (`end_time < now`), so old test/seed events don't linger forever; and events gained a hard-delete path (`DELETE /business/venues/:id/events/:eventId`, `DELETE /neighborhood-admin/neighborhoods/:id/events/:eventId`) with an ownership check against the event's actual `venue_id`/`neighborhood_id`, for cleaning up a bad manual entry or import without waiting on the fuller edit/hide Events tab (BACKLOG.md Ref 78). `event` also gained a `location` column, populated only for imported events: a neighborhood-owned import keeps the feed's own per-VEVENT `LOCATION` as-is (a neighborhood-wide feed's events can be anywhere), while a venue-owned import overwrites it with that venue's own address (`icalSync.ts`'s `syncVenueIcalFeed`, falling back to the feed's LOCATION only if the venue has no address on file) since a business's events are always at that business. Manually-created events still leave `location` null. `event` also gained a `status` column (`active`|`hidden`, default `active`) with a new `PATCH .../events/:eventId/status` route (same ownership check/gating as the delete route): unlike a hard delete, hiding an event survives a future "Sync now" — `upsertImportedEvents` deliberately never includes `status` in its upsert payload, so `ON CONFLICT DO UPDATE` leaves an already-hidden row's status untouched and only a brand-new row gets the column default. `listEventsForNeighborhoodAndVenues` (public Upcoming tab/Happening now) and `listEventsForVenue`'s public caller (`GET /venues/:id/events`) now exclude hidden rows; the admin dashboards (which need to show hidden events so there's something to unhide) pass `includeHidden = true`. Both admin dashboards gained Hide/Unhide buttons and a "Hidden" badge per event.
- **2026-07-15** — Super admin role (BACKLOG.md): a new rung above `adminGate`'s "admin of at least one neighborhood," implemented the same way as `neighborhood_admin` — a plain grant table (`super_admin`, `user_id` unique, no role column) rather than a boolean column on `app_user`, checked via `SuperAdminRepository.isSuperAdmin` and a new `requireSuperAdmin` gate, granted via a new CLI script (`npm run grant:super-admin -- <email>`, mirroring `grant:admin`). Two effects: (1) `POST /admin/neighborhoods` (creating a brand-new neighborhood) moved from `adminGate` to `superAdminGate` — restricted for now, while the platform is small, not a permanent limitation; the dry-run `preview-boundary` route stays on `adminGate` since it's shared with the existing-neighborhood boundary-redraw flow. `admin/neighborhood/new/page.tsx` gained a client-side super-admin check (forbidden message) rather than only relying on the API's 403; the "+ New neighborhood" entry points (`AdminSwitcher.tsx`, `admin/page.tsx`'s empty state) now gate on `is_super_admin` instead of `is_neighborhood_admin`. (2) A super admin bypasses the 24h "Reimport Locations" cooldown (`getLocationsReviewCooldownStatus` gained a `bypassCooldown` param) on both `GET .../locations/review` and `.../locations/review/status`, while still seeing the real last-reviewed/next-allowed times. New `AppUser.is_super_admin` field, computed the same way as `is_neighborhood_admin` (derived per-request in `toAppUser`, not stored on the user record itself).
- **2026-07-15** — "Reimport Locations" (BACKLOG.md): the Locations tab gained a prominent button linking into the existing bulk Places review wizard, rate-limited to once per 24h per neighborhood (`neighborhood.locations_reviewed_at`, checked and stamped server-side, not just a disabled button — new `GET .../locations/review/status` read-only endpoint backs both pages' cooldown display, and the real `GET .../locations/review` 429s with `next_allowed_at` if called early).
